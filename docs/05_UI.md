# 05_UI.md

## UI実装仕様

## 1. 結論
UIは「候補確認 → AI下書き → 人間レビュー → 承認 → 送信 → 返信対応」を迷わず進める業務画面として実装する。

MVPでは追加フロントエンドを立てず、NestJS API内にローカル営業画面を同梱する。API起動後、`http://localhost:3000/` で利用する。

ローカル営業画面は、実際の営業業務順に合わせて以下の5ステップで構成する。

1. 取り込む: CAMPFIRE URLから案件を登録する。
2. 選ぶ: 営業リストから優先して対応する案件を選ぶ。
3. 見る: 案件詳細、商品特徴、支援状況を確認する。
4. 作る: AI分析を見て、営業メール下書きを生成する。
5. 確認する: 本文編集、送信前チェック、レビュー依頼、承認、キュー投入を行う。

## 3. Lead一覧
カラム: Priority、Score、Company、Project、Platform、Amount、Supporters、Status、NextAction、LastMail。Filter: status、priority、platform、score、amount、supporterCount、hasReply、hasClick。

MVPのCAMPFIRE取り込みでは、会社名・案件名・URL・商品名の曖昧検索、カテゴリ検索、現在公開中、終了間近、支援額レンジ、サポーター数レンジ、過去プロジェクト数レンジでCAMPFIRE候補URLを検索できる。これは営業リストを絞り込むためではなく、営業リストへ取り込む前の候補探しに使う。カテゴリはCAMPFIREから取得した表示名とカテゴリURLに、基本カテゴリ候補を加えてプルダウン表示する。CAMPFIREカテゴリURLがある項目はカテゴリURLを使い、基本カテゴリは検索キーワードとして使う。支援額、サポーター数、過去プロジェクト数は、毎回手入力せずに候補レンジから選べるようにする。

MVPの `/` 画面では、左側に取り込みと営業リスト、右側にCAMPFIRE検索結果、選択案件の詳細、AI分析、メール確認を表示する。CAMPFIRE検索結果は右側にカード形式で表示し、候補URL、支援額、サポーター数、残り日数、取り込み操作を確認できる。右側の案件情報は `案件詳細`、`AI分析`、`メール確認` のタブで切り替える。上部には業務ステップを表示し、どの順番で操作するかを明確にする。

営業リストでは、まず以下を表示する。

- 会社名
- 案件名
- Leadステータス
- Score
- Priority
- Lead reason

`GET /api/leads` は画面表示用に `company`、`project`、最新 `scores` を含めて返す。

営業リストでLeadを選択した場合、同じ画面に案件詳細パネルを表示する。

- 企業名
- 案件名
- CAMPFIRE URL
- カテゴリ
- 支援額
- 支援者数
- Lead状態
- Priority
- Score
- Lead reason
- Project description
- 連絡先確認: メールアドレス、問い合わせフォームURL、サイト内メッセージURL、連絡先メモ
- 送信記録: 送信手段、送信日、次回確認日
- ブランド/SNS分析: 公式サイト、Instagram、TikTok、X、ブランド分析メモ、SNS分析メモ

案件詳細パネルにはCAMPFIRE URLを開くボタンを置く。

CAMPFIRE取り込み時は、ページ内リンクから公式サイト、問い合わせフォーム、Instagram、TikTok、Xを可能な範囲で自動取得し、営業管理欄の空欄に反映する。手入力済みのURLは自動取得で上書きしない。

過去プロジェクトが100件以上ある可能性がある実行者は、過去案件の詳細スクレイピング対象から外し、案件詳細に注意表示を出す。
過去プロジェクト数は、実行者プロフィールページの「他にN件のプロジェクトを掲載しています」という表示を優先して取得する。

営業リストでLeadを選択した場合、AI分析結果パネルも表示する。

- 最新AI生成日時
- AI種別
- 使用モデル
- 使用トークン
- factsUsed
- assumptions
- riskFlags
- 生成件名
- 生成本文プレビュー
- AI生成履歴

## 4. Lead詳細
表示: Summary、Company、Project、LeadScore内訳、Mail Timeline、Reply Timeline、Tasks、AiGeneration、AuditLog。

## 5. Mail Review
承認者はTo、件名、本文、会社NG、配信停止、AI riskFlags、添付URLを確認する。Approveはmanager以上かつ `in_review` のみ可能。

MVPの `/` 画面では、生成済みメール一覧、件名、本文編集、保存、レビュー依頼、棄却、再レビュー依頼、承認、キュー投入を同一画面で行う。

`GET /api/mails` は画面表示用に `company` と `lead.project` を含めて返す。

送信前チェックリストはメール確認エリア内に表示し、メールごとにDB保存する。各チェック項目は行全体をクリックできる大きさで表示する。選択中メールのチェック状態を読み込み、チェック変更時に保存する。チェック済み項目を上、未チェック項目を下に表示し、チェックを外した項目は一番下の未チェック側へ移動する。全項目が完了していない場合、承認とキュー投入はできない。

メールはレビュー依頼後に棄却できる。メールを棄却した場合、棄却理由をメール確認エリアに表示する。担当者は件名・本文を修正して保存したあと、「再レビュー依頼」を押すことで `rejected -> in_review` に戻せる。

## 6. Reply Inbox
分類: interested、need_info、meeting_request、not_interested、unsubscribe、complaint、auto_reply、unknown。unsubscribe/complaintは強制的にmanager確認を表示。

## 7. 共通コンポーネント
StatusBadge、PriorityBadge、ScoreBar、ConfirmDialog、AuditTimeline、DiffViewer、MarkdownPreview、ErrorAlertを作る。

MVPのローカル営業画面では、DB/APIのenum値は保持したまま画面表示だけ日本語化する。

- Mail status: draft=下書き、in_review=確認待ち、rejected=棄却、approved=承認済み、queued=送信待ち、sending=送信中、sent=送信済み、failed=送信失敗、cancelled=キャンセル
- Lead status: discovered=発見、qualified=候補、drafted=下書き済み、reviewing=確認中、approved=承認済み、queued=送信待ち、contacted=連絡済み、replied=返信あり、meeting_candidate=商談候補、rejected=対象外、no_response=返信なし、archived=アーカイブ
- Priority: high=高、medium=中、low=低

## 8. Validation
emailは形式チェック、URLはhttp/https、amount/supporterCountは0以上、subject/bodyは空不可。

## 9. 完了条件
Lead一覧からAI下書き生成、レビュー依頼、承認、返信確認まで画面操作できる。

MVP Phase 1の完了条件:

- `/` でローカル営業画面を開ける。
- 画面上部で `取り込む -> 選ぶ -> 見る -> 作る -> 確認する` の業務順を確認できる。
- CAMPFIREからカテゴリ候補を取得し、検索条件としてプルダウン表示できる。
- CAMPFIRE候補検索で、曖昧検索、カテゴリ、現在公開中、終了間近、支援額レンジ、サポーター数レンジ、過去プロジェクト数レンジを指定して取り込み候補URLを探せる。
- CAMPFIRE候補検索の結果を右側の検索結果エリアに表示できる。
- CAMPFIRE候補検索の結果から、選択したURLを営業リストへ取り込める。
- CAMPFIRE URLを取り込める。
- 取り込み結果が営業リストに表示される。
- 営業リストで選択した案件の詳細情報を確認できる。
- 営業リストで選択した案件の連絡先、送信記録、ブランド/SNS分析メモを保存できる。
- CAMPFIRE取り込み時に公式サイト、問い合わせフォーム、SNS URLを可能な範囲で自動取得できる。
- 過去プロジェクトが100件以上ある可能性がある実行者に注意表示を出せる。
- 右側の詳細、AI分析、メール確認をタブで切り替えられる。
- 営業リストで選択した案件のAI分析・生成履歴を確認できる。
- 選択LeadからAIメール下書きを生成できる。
- 件名と本文を編集して保存できる。
- 送信前チェックリストを保存できる。
- `draft -> in_review -> approved -> queued`、`in_review/approved -> rejected`、`rejected -> in_review` を画面操作できる。
- 未承認メール、または送信前チェック未完了メールはキュー投入できない。

## Codex実装条件
- docs/ を正本として実装する。
- DB変更時は `docs/07_DATABASE.md` と `prisma/schema.prisma` を同時更新する。
- API変更時は `docs/06_API.md` と `openapi/openapi.yaml` を同時更新する。
- AI生成・メール送信・承認・配信停止は必ずログを残す。
- MVPではAI生成メールの自動送信は禁止。人間承認を必須にする。
