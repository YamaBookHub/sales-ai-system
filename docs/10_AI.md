# 10_AI.md - AI機能 実装仕様

## 1. 目的
営業AIシステムにおけるAI機能は、営業候補の評価、メール文面生成、返信分類、次アクション提案を支援する。AIは最終判断者ではなく、営業担当者の判断を補助する。

## 2. 適用範囲
- クラウドファンディング案件の要約
- 営業候補スコアリング
- 初回営業メール生成
- 件名生成
- リマインド文生成
- 返信分類
- 温度感判定
- 次アクション提案
- 営業メモの構造化

## 3. 非適用範囲
- 人間確認なしの完全自動送信
- 法的判断の自動化
- 事実確認なしの実績生成
- 存在しない商品特徴の創作
- 相手企業に関する未確認断定

## 4. 基本方針
AI出力は必ず `AiGeneration` に保存する。保存対象は、入力JSON、出力JSON、モデル名、プロンプトバージョン、遅延、トークン数、コストである。営業送信に使う文章は、必ずレビュー状態を経由する。

## 6. Prompt管理
プロンプトはコード直書き禁止。`src/ai/prompts/*.ts` にバージョン付きで定義する。

```ts
export const PROMPT_VERSION = 'sales-ai-v1.0.0';
```

### 6.1 共通System Prompt
- 事実と推測を分ける
- 不明点を補完しない
- 営業感を抑える
- 実績を創作しない
- 相手情報から確認できる内容だけを使う
- 出力はJSON Schemaに従う

### 6.2 メール生成Prompt入力
```json
{
  "company": {"name":"string","industry":"string|null","websiteUrl":"string|null"},
  "project": {"title":"string","url":"string","amount":0,"supporterCount":0,"category":"string|null"},
  "service": {"companyName":"株式会社第弐ヴォヌール","offer":"クラウドファンディング支援/SNSマーケティング支援"},
  "tone": "low_sales_pressure",
  "templateKey": "initial_cf_outreach_v1"
}
```

### 6.3 メール生成Prompt出力
```json
{
  "subject": "string",
  "bodyText": "string",
  "factsUsed": ["string"],
  "assumptions": ["string"],
  "riskFlags": ["string"],
  "cta": "string"

```

## 7. JSON Schema制約
AI出力は自由文のまま使わない。Zodで検証し、失敗時は `AI_OUTPUT_SCHEMA_INVALID` として失敗扱いにする。

```ts
const GeneratedMailSchema = z.object({
  subject: z.string().min(5).max(80),
  bodyText: z.string().min(200).max(3000),
  factsUsed: z.array(z.string()).default([]),
  assumptions: z.array(z.string()).default([]),
  riskFlags: z.array(z.string()).default([]),
  cta: z.string().max(300),
});
```

## 8. スコアリング仕様
AIはスコアを直接確定しない。ルールベース点数を主とし、AIは理由説明と補助判定に限定する。

### 8.1 ルールベース点数
| 項目 | 条件 | 点数 |
|---|---|---:|
| amountScore | 支援金額100万円未満 | +25 |
| supporterScore | サポーター50人未満 | +25 |
| urgencyScore | 終了まで14日未満 | +15 |
| fitScore | 商品/物販/ガジェット/生活雑貨 | +20 |
| activityScore | 更新やSNS導線が弱い | +15 |

合計100点。`LeadScore.totalScore` に保存する。

### 8.2 優先度変換
| totalScore | priority |
|---:|---|
| 70以上 | high |
| 40〜69 | medium |
| 39以下 | low |

## 9. 返信分類仕様
返信本文を以下に分類する。

| category | 意味 | 後続処理 |
|---|---|---|
| interested | 関心あり | task作成 |
| need_info | 追加情報希望 | 返信案作成 |
| meeting_request | 日程調整希望 | meeting_candidate |
| not_interested | 興味なし | rejected |
| unsubscribe | 配信停止 | contact/company停止 |
| auto_reply | 自動返信 | 状態維持 |
| complaint | 苦情 | 送信停止・管理者確認 |
| unknown | 判定不能 | 人間確認 |

## 10. 安全設計
- 送信前に人間レビュー必須
- `unsubscribe` と `complaint` は自動で送信停止
- AIが生成した本文は `draft` から開始
- `approved` 以外は送信キューに入れない
- 実績、数値、日付は入力に存在するものだけ使用

## 14. テスト観点
- JSON Schema不一致時に失敗する
- 存在しない実績を本文に入れない
- unsubscribeを正しく判定する
- complaint時に送信停止する
- token/cost/latencyが保存される
- promptVersionが保存される

## 15. Codex実装指示
CodexはAI本文を直接送信する処理を実装してはいけない。必ず `OutreachEmail.status=draft` に保存し、レビューAPIを経由して `approved` にする。AI出力はすべて `AiGeneration` に保存し、後から再現できる構造にする。
